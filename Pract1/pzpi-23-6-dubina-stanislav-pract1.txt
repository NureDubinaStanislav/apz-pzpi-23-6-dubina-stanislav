2 ЗАВДАННЯ

     Метою практичного заняття є ґрунтовне засвоєння фундаментальних принципів об’єктно-орієнтованого проєктування через дослідження класичних шаблонів розробки програмного забезпечення. Студент має опанувати класифікацію GoF (Gang of Four), навчитись аналізувати архітектурні властивості обраного патерна, його цільове призначення, умови застосування, а також сильні та слабкі сторони у контексті побудови складних програмних систем. У межах виконання роботи студент самостійно обирає один із патернів (окрім Singleton) з праці Е. Ґамми та співавторів «ООП. Патерни проєктування», здійснює його теоретичне дослідження та виконує практичну реалізацію погодженою з викладачем мовою програмування.
     Обов’язковим складником роботи є розробка розширеної мультимедійної презентації та відеодоповіді тривалістю 10–12 хвилин, яка охоплює структуру шаблона, UML-діаграми взаємодії компонентів і демонстрацію програмного коду. Відеозапис має бути оприлюднений на платформі YouTube з дотриманням вимог щодо мови подачі матеріалу та хронологічного опису фрагментів доповіді.
     Підсумком виконання практичного заняття є оформлення фінального звіту відповідно до вимог ДСТУ 3008:2015. Звіт повинен містити детальний опис проведеної роботи, висновки й додатки з ілюстративними матеріалами (слайдами) та програмним кодом у текстовому форматі з нумерацією рядків. Матеріали подаються у форматах PDF та TXT (UTF-8) для розміщення на навчальній платформі dl.nure.ua й у відповідному репозиторії GitHub. Під час захисту студент зобов’язаний продемонструвати розуміння обраної архітектурної моделі та здатність аргументувати доцільність її застосування для конкретних проєктних завдань.
      	

3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблони проєктування є перевіреними та систематизованими підходами до розв’язання поширених архітектурних проблем, що виникають у процесі розробки програмних систем. На відміну від готового коду, патерни являють собою узагальнені описи того, яким чином слід організовувати структуру класів та їхню взаємодію для досягнення конкретних архітектурних цілей. Їх головна цінність полягає у забезпеченні гнучкості, розширюваності та довготривалої підтримуваності програмного продукту, а також у формуванні єдиного професійного словника між учасниками команди розробників.
     Класична систематизація, запропонована групою Gang of Four (GoF) у 1994 році, поділяє всі 23 патерни на три принципові групи залежно від характеру задачі, яку вони вирішують. Першу групу утворюють породжувальні (Creational) шаблони, спрямовані на управління механізмами створення об’єктів та забезпечення незалежності системи від способу їх інстанціювання. До другої групи належать структурні (Structural) шаблони, що описують принципи організації класів та об’єктів у більш складні структури при збереженні їхньої ефективності. Третя група — поведінкові (Behavioral) шаблони — охоплює алгоритми взаємодії між об’єктами та механізми розподілу відповідальностей між ними.
     Патерн Decorator, що є предметом цього звіту, належить до категорії структурних шаблонів і вирізняється серед них можливістю динамічного нарощування функціональності об’єктів без модифікації їхнього вихідного коду та без побудови розгалужених ієрархій спадкування. 
     3.2 Опис обраного патерна
     Назва патерна: Decorator (Декоратор).
     Тип шаблона (патерна): структурний (Structural).
     Основне призначення: динамічне додавання нових обов’язків об’єкту шляхом його обгортання в об’єкти-декоратори, які реалізують той самий інтерфейс. Патерн слугує гнучкою альтернативою механізму спадкування для розширення поведінки, дозволяючи комбінувати функціональність під час виконання програми без зміни вихідного коду базового компонента.
     Передумови застосування: шаблон доцільно використовувати, коли розширення функціональності через спадкування призводить до «вибуху класів» — експоненційного зростання їх кількості при збільшенні кількості комбінацій властивостей. Наприклад, для об’єкта «кава» з можливими добавками (молоко, цукор, вершки) механізм спадкування породжує окремий клас для кожної комбінації, тоді як Decorator вирішує цю проблему обгортанням об’єктів у довільному порядку під час виконання.
     Переваги:
     1) Розширення функціональності без модифікації існуючого коду, що відповідає принципу відкритості/закритості (Open/Closed Principle);
     2) Гнучке комбінування різних поведінок у довільних поєднаннях під час виконання програми;
     3) Дотримання принципу єдиної відповідальності (Single Responsibility Principle) — кожен декоратор відповідає лише за одну додаткову функцію;
     4) Можливість динамічно додавати та відкликати обов’язки об’єктів без втручання у вихідний код.
     Недоліки:
     1) Ускладнення процесу налагодження через наявність великої кількості дрібних об’єктів-обгорток;
     2) Суттєва залежність кінцевого результату від порядку накладання декораторів;
     3) Технічна складність видалення декоратора з довільного місця ланцюжка обгорток;
     4) Значне збільшення кількості малих класів у системі, що ускладнює її загальне розуміння. 
     3.3 Структура шаблона (патерна)
     Опис структурних частин:
     Архітектура патерна Decorator побудована на принципі делегування та єдиному інтерфейсі, що забезпечує взаємозамінність компонентів і декораторів у клієнтському коді. Основними структурними елементами є:
     1) Component (Компонент): спільний інтерфейс, який реалізують як базовий об’єкт, так і всі декоратори. Завдяки цьому клієнтський код може однаково взаємодіяти з будь-яким рівнем обгортання.
     2) ConcreteComponent (Конкретний компонент): базова реалізація інтерфейсу Component, що є первинним об’єктом, поведінку якого планується розширювати шляхом декорування.
     3) Decorator (Декоратор): абстрактний клас або інтерфейс, що також реалізує Component та зберігає всередині посилання на загорнутий об’єкт типу Component. Усі виклики методів делегуються до цього внутрішнього об’єкта.
     4) ConcreteDecoratorA / ConcreteDecoratorB (Конкретні декоратори): підкласи Decorator, кожен з яких визначає специфічну додаткову поведінку, що накладається поверх делегованого виклику до обгорнутого об’єкта.
     Опис взаємодії об’єктів:
     Механізм роботи патерна базується на ланцюговому делегуванні викликів. Клієнт звертається до зовнішнього декоратора, який виконує власну додаткову логіку і передає виклик методу до наступного вкладеного об’єкта. Цей процес рекурсивно повторюється аж до досягнення ConcreteComponent, після чого результат повертається у зворотному напрямку — кожен декоратор у ланцюжку може доповнити або модифікувати відповідь перед поверненням клієнту.
     Послідовність взаємодії при виклику operation() виглядає таким чином: Client викликає DecoratorB, DecoratorB делегує виклик DecoratorA, DecoratorA делегує виклик ConcreteComponent. Результат повертається у зворотному напрямку: ConcreteComponent → DecoratorA → DecoratorB → Client.
     UML-діаграми з поясненнями:
     Для унаочнення архітектури патерна Decorator використовується діаграма класів, що відображає ієрархічну структуру компонентів, відносини реалізації та агрегації між ними.

Рисунок 3.1 – UML-діаграма класів патерна Decorator
     Пояснення до структури діаграми:
     Інтерфейс Component встановлює спільний контракт для ConcreteComponent та Decorator, що уможливлює прозору взаємозаміну у клієнтському коді.
     Клас Decorator містить поле типу Component (агрегація), через яке відбувається делегування. Відносини агрегації (1 до 0..*) між Decorator та Component означають, що один декоратор може утримувати довільний ланцюжок вкладених компонентів.
     Конкретні декоратори ConcreteDecoratorA та ConcreteDecoratorB успадковують Decorator і перевизначають методи, додаючи власну поведінку до делегованого виклику.
     Діаграма послідовності (Sequence Diagram) ілюструє порядок передачі виклику operation() крізь ланцюжок декораторів та зворотне повернення результату.
     3.4 Реалізація шаблона (патерна)
     Практичну реалізацію патерна Decorator виконано мовою програмування Java. Вибір мови обумовлений строгою типізацією та підтримкою механізму інтерфейсів, що є ключовим для коректної побудови архітектури цього шаблона. Повний програмний код наведено у Додатку В.
     Визначення інтерфейсу та базових класів:
     На першому етапі розробки було визначено спільний інтерфейс Coffee з методами getDescription() та getCost(), який виконує роль контракту Component. На його основі реалізовано клас SimpleCoffee як базовий ConcreteComponent з вартістю 10.0 та описом «Проста кава». Абстрактний клас CoffeeDecorator реалізує інтерфейс Coffee і зберігає посилання decoratedCoffee на обгорнутий об’єкт, делегуючи йому всі виклики за замовчуванням. Такий підхід забезпечує дотримання принципу інверсії залежностей.
     Реалізація конкретних декораторів:
     На наступному кроці було розроблено конкретні декоратори MilkDecorator та SugarDecorator, що успадковують CoffeeDecorator. Кожен з них перевизначає методи getDescription() та getCost(), додаючи до результату делегованого виклику власні значення: молоко збільшує вартість на 3.0 та доповнює опис рядком «, молоко», цукор — на 1.5 зі значенням «, цукор».
     Демонстрація використання:
Для перевірки коректності реалізації було сформовано об’єкт Coffee шляхом послідовного обгортання: new SugarDecorator(new MilkDecorator(new SimpleCoffee())). Виклик методів getDescription() та getCost() повернув відповідно «Проста кава, молоко, цукор» та 14.5, що підтверджує правильну роботу ланцюжка декораторів. Ключова перевага підходу полягає у тому, що будь-яка нова комбінація добавок формується без написання нових класів — лише шляхом зміни порядку обгортання.

4 ВИСНОВКИ

     У процесі виконання практичного заняття було всебічно розглянуто архітектуру структурного шаблона проєктування Decorator (Декоратор) та виконано всі передбачені завданням етапи роботи. Зокрема, досліджено призначення патерна та визначено умови, за яких його застосування є обґрунтованим — насамперед у ситуаціях, коли спадкування призводить до комбінаторного вибуху кількості класів. Вивчено переваги патерна, серед яких — відповідність принципу Open/Closed, динамічне управління поведінкою та підтримка принципу єдиної відповідальності.
     Розроблено практичну реалізацію патерна мовою Java на прикладі системи замовлення кави. В реалізації задіяно інтерфейс Coffee, базовий клас SimpleCoffee, абстрактний декоратор CoffeeDecorator та конкретні декоратори MilkDecorator і SugarDecorator. Результати виконання коду підтвердили коректність роботи ланцюжка обгорток та правильність обчислення сукупної вартості замовлення. Для наочного відображення структури архітектурного рішення побудовано UML-діаграму класів та діаграму послідовності взаємодії компонентів.
     Також підготовлено мультимедійну презентацію та відеодоповідь для платформи YouTube, що охоплює теоретичне обґрунтування патерна, розгляд реальних прикладів його застосування (Java I/O Streams, Python-декоратори, middleware в Express.js та Django) та демонстрацію написаного коду.
   Результатом роботи стало оформлення підсумкового звіту відповідно до вимог ДСТУ 3008:2015. Виконання завдання сприяло формуванню практичних навичок проєктування розширюваних програмних систем, поглибленому розумінню принципів SOLID, а також набуттю досвіду ведення технічної документації. Отримані знання підтверджують ефективність шаблонів GoF як інструменту для побудови масштабованого та якісного програмного забезпечення.

8           return "Проста кава";
9       }
10  
11      public double getCost() {
12          return 10.0;
13      }
14  }
15  
16  abstract class CoffeeDecorator implements Coffee {
17      protected Coffee decoratedCoffee;
18  
19      public CoffeeDecorator(Coffee coffee) {
20          this.decoratedCoffee = coffee;
21      }
22  
23      public String getDescription() {
24          return decoratedCoffee.getDescription();
25      }
26  
27      public double getCost() {
28          return decoratedCoffee.getCost();
29      }
30  }
31  
32  class MilkDecorator extends CoffeeDecorator {
33      public MilkDecorator(Coffee coffee) {
34          super(coffee);
35      }
36  
37      public String getDescription() {

39      }
40

42          return decoratedCoffee.getCost() + 3.0;
43      }
44  }
