3 ОПИС ВИКОНАНОЇ РОБОТИ

 Загальні відомості про систему
     Netflix — одна з найбільших платформ потокового відео у світі. Компанія була заснована у 1997 році як сервіс прокату DVD-дисків поштою, а у 2007 році здійснила перехід до моделі потокового відео через Інтернет, що кардинально змінило індустрію розваг.
     Станом на 2025 рік платформа налічує понад 260 мільйонів підписників у 190+ країнах світу. Щодня користувачам доступно понад 15 000 годин контенту. У пікові вечірні години Netflix генерує близько 37% інтернет-трафіку США. Річний дохід компанії за підсумками 2023 року перевищив 33 мільярди доларів США. Ці показники свідчать про надвисокі вимоги до надійності та масштабованості інфраструктури платформи.
 Архітектурний стиль системи
     Архітектура Netflix пройшла значну еволюцію. У 2007–2008 роках система являла собою єдину монолітну Java-систему з базою даних Oracle та ручним розгортанням. У 2008 році стався критичний збій — база даних була недоступна впродовж трьох діб, що стало поштовхом до повного реінжинірингу системи. У 2009–2012 роках розпочалася поступова міграція на AWS та розбиття моноліту на незалежні мікросервіси. У 2016 році завершився повний перехід на AWS та побудова сучасної хмарної інфраструктури. Еволюція від монолітної до мікросервісної архітектури зайняла майже 10 років і стала еталоном для всієї індустрії.
     Сьогодні Netflix використовує мікросервісну архітектуру (Microservices Architecture). Ключові принципи: понад 1000 незалежних мікросервісів у продакшені; кожен сервіс — окрема команда розробки відповідно до закону Конвея; loose coupling та high cohesion — слабка зв’язність між сервісами та висока внутрішня згуртованість; API-first підхід та zero-downtime deployment; event-driven комунікація через Apache

Kafka. Мікросервісна архітектура була обрана насамперед для забезпечення незалежного масштабування окремих компонентів, швидкого розгортання нових функцій без зупинки системи та ізоляції збоїв — падіння одного сервісу не впливає на роботу інших.
 Основні компоненти системи
     Архітектура Netflix складається з кількох рівнів, кожен з яких відповідає за окрему зону відповідальності.
     Client Layer — рівень клієнтських додатків. Включає мобільні застосунки для iOS та Android, веббраузери, Smart TV та ігрові консолі — точки доступу користувачів до сервісу.
     API Gateway (Zuul) — єдина точка входу для всіх запитів. Забезпечує маршрутизацію до відповідних мікросервісів, rate limiting та первинну автентифікацію.
     Мікросервіси — основний обчислювальний рівень: рекомендації, пошук, авторизація, білінг, каталог — понад 1000 незалежних сервісів.
     Messaging Layer (Apache Kafka) — асинхронна комунікація між сервісами, обробка 700 млрд подій на день.
     Data Layer — рівень зберігання даних, реалізований за підходом поліглотної персистентності: Cassandra, MySQL, EVCache, S3 — різні бази для різних задач.
     CDN Open Connect — власна мережа доставки відеоконтенту у 1000+ містах по всьому світу.
 Взаємодія компонентів
     Розглянемо типовий сценарій — запуск відтворення відео користувачем. Користувач натискає кнопку «Play», після чого клієнт надсилає HTTPS-запит до API Gateway Zuul. Zuul перенаправляє запит до Auth Service для перевірки токена авторизації. Після успішної автентифікації Playback Service формує відповідь, звертаючись до License Service  для  отримання  DRM-ключів  захисту  контенту.  Клієнт  отримує

URL відеосегментів від найближчого вузла Open Connect та розпочинає стримінг за протоколом MPEG-DASH або HLS. Повний цикл запуску відео від натискання кнопки до початку стримінгу охоплює авторизацію, DRM-захист та оптимальний вибір CDN-сервера.
     Між мікросервісами використовуються два типи взаємодії: синхронна через REST/HTTP для запитів, що потребують негайної відповіді, та асинхронна через Apache Kafka для подієво-орієнтованих потоків даних (перегляди, кліки, аналітика).
 Масштабованість та надійність
     Netflix розгорнутий на платформі Amazon Web Services. Використовуються  такі  сервіси:  EC2  —  обчислювальні  потужності для мікросервісів; S3 — зберігання медіаконтенту та резервних копій; DynamoDB — сесії користувачів та рейтинги; Route 53 — DNS та геобалансування навантаження. Система розгорнута у трьох основних регіонах — us-east-1, eu-west-1, ap-northeast-1 — в конфігурації Active-Active, де всі регіони одночасно обслуговують трафік. Автоматичний failover забезпечує перемикання між регіонами при збоях, а цільовий час відновлення (RTO) складає менше 30 секунд.
Ключовим підходом до забезпечення надійності є Chaos Engineering
— практика навмисного руйнування системи у продакшені для виявлення слабких місць. Набір інструментів Simian Army включає: Chaos Monkey
— випадково вимикає інстанси у продакшені для перевірки стійкості системи; Latency Monkey — штучно додає затримки у мережеву комунікацію між сервісами; Security Monkey — автоматично шукає вразливості та неправильні конфігурації безпеки. Завдяки постійному тестуванню в умовах реальних збоїв Netflix досягає рівня доступності 99,99% при мільйонах одночасних користувачів.
     Для зберігання даних Netflix застосовує підхід Polyglot Persistence: Apache  Cassandra  —  профілі  користувачів,  мільярди  записів;  EVCache

(Memcached) — 30 млн операцій за секунду, затримка менше 1 мс; Apache Kafka — 700 млрд подій на день; Elasticsearch — повнотекстовий пошук по каталогу.
     Рекомендаційний двигун є основою успіху Netflix, забезпечуючи понад 80% всіх переглядів контенту. В продакшені одночасно працює понад 300 ML-моделей, а паралельно проводиться понад 1000 A/B тестів.

     У межах практичного заняття підготовлено чотири приклади програмного коду мовою Python, що демонструють ключові архітектурні принципи системи Netflix. Лістинг В.1 демонструє алгоритм вибору CDN-сервера Open Connect: алгоритм враховує географічну близькість та поточне навантаження на сервер для оптимального вибору. Лістинг В.2 показує взаємодію клієнта з Playback API при запуску відео. Лістинг В.3 містить реалізацію Chaos Monkey — планового агента, що щогодини з певною ймовірністю завершує роботу випадкового інстансу, змушуючи систему адаптуватися в реальному часі. Лістинг В.4 демонструє доступ до бази даних Apache Cassandra.

4 ВИСНОВКИ

     У процесі виконання практичного заняття було всебічно досліджено архітектуру програмної системи Netflix та виконано всі передбачені завданням етапи роботи. Зокрема, досліджено еволюцію системи від монолітної Java-системи до платформи з понад тисячею незалежних мікросервісів, що зайняла майже 10 років та стала еталоном для всієї індустрії.  Визначено  мікросервісний  архітектурний  стиль  системи та обґрунтовано причини його вибору — незалежне масштабування, швидке розгортання та ізоляція збоїв. Проаналізовано взаємодію ключових компонентів: API Gateway Zuul, мікросервісів, Apache Kafka, CDN Open Connect та рівня зберігання даних. Оцінено механізми забезпечення масштабованості та надійності, зокрема підхід Chaos Engineering та Polyglot Persistence, що дозволяє досягати 99,99% доступності при мільйонах одночасних користувачів. Підготовлено мультимедійну презентацію та відеодоповідь. Сформовано практичні навички архітектурного аналізу складних програмних систем.
